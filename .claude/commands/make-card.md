---

# 역할

사용자가 작성한 학습 노트(`c-lang/`, `stm32/`, `cs/` 하위 `.md`)를 읽고, **각 핵심 개념마다 답변카드 1개씩** 생성합니다.

답변카드는 `면접대비/답변카드/{개념명}.md` 로 저장됩니다.

---

# 입력

- 인자: 학습 노트 파일 경로 (예: `c-lang/포인터.md`)
- 인자 없으면 → 사용자에게 "어떤 학습 노트로 만들까?" 묻기
- 인자가 폴더면 → 폴더 내 모든 `.md` 파일 일괄 처리 여부 확인

---

# 동작 순서

## 1. 학습 노트 분석

- 파일을 읽고 **개념 단위로 분해**:
  - `## 1. 제목`, `## 2. 제목` 식으로 번호 붙은 큰 섹션 → 각각 별도 카드
  - `### 하위제목` 도 독립 개념이면 별도 카드 가능
  - 너무 작은 개념은 1개 카드로 묶음
- 추출 단위 판단 기준:
  - 면접에서 "X가 뭐예요?" 라고 단독 질문할 만한가?
  - 30초 안에 설명 가능한 분량인가?

## 2. 추출 계획 제시 (사용자 확인)

먼저 카드 후보 목록을 사용자에게 보여주고 동의 받기:

```
c-lang/포인터.md 분석 결과:

생성할 답변카드 후보:
1. 포인터.md (## 1. 포인터란)
2. 포인터연산.md (## 2. 포인터 연산)
3. 포인터배열.md (## 3. 포인터와 배열)
4. 이중포인터.md (### 이중 포인터 — 심화)

이대로 진행할까? 빼거나 합칠 거 있어?
```

사용자가 OK 하면 진행. 수정 요청 있으면 반영 후 다시 확인.

## 3. 각 카드 생성

`templates/답변카드.md` 를 기준 골격으로 사용. 각 섹션 채우는 규칙:

### 30초 통합 답변
- 학습 노트의 본문을 면접용으로 압축
- 한 단락, 자연스러운 한국어
- "음…", "그거", "일케" 같은 모호어 절대 금지
- 정확한 기술 용어 사용 (예: "값이 안 바뀜" → "컴파일러 최적화로 캐싱됨")

### 한 줄 정의
- `{개념}은 {무엇}이다. {핵심 특성}.` 형식 엄수
- 학습 노트의 첫 문단/콜아웃에서 추출

### 왜 필요한가
- 학습 노트의 도입부 / "왜" 섹션에서 추출
- 1~2문장

### 동작 원리
- 메모리/타이밍/하드웨어 동작 위주
- 코드는 넣지 않음 (말로 설명하는 분량)

### 비교 / 대안
- 학습 노트에 비교 대상 있으면 그것
- 없으면 일반적 짝 개념 제시 (예: `static` vs `extern`, `volatile` 있음 vs 없음)

### 꼬리질문 예상
- 임베디드 면접에서 실제 자주 나오는 질문 위주
- 2~3개

### 키워드
- 그 개념 설명할 때 입에서 나와야 할 단어 3~5개

---

## 4. 저장 규칙

- 파일 경로: `면접대비/답변카드/{개념명}.md`
- 파일명은 학습 노트의 섹션 제목 기반 (공백 → 언더스코어, 특수문자 제거)
- **이미 존재하면**:
  - 사용자에게 알림: "{파일} 이미 있음. 덮어쓸까 / 병합할까 / 건너뛸까?"
  - 기본은 **병합** (기존 사용자 수정 보존, 빈 섹션만 채움)
- 출처 헤더에 원본 학습 노트 경로 자동 기입
- 최종 갱신일 = 오늘 날짜

---

## 5. 완료 보고

```
✓ 생성 완료: 면접대비/답변카드/
  - 포인터.md (신규)
  - 포인터연산.md (신규)
  - 포인터배열.md (병합, 30초답변만 갱신)

다음 단계 추천:
- /easy-quiz 포인터 — 바로 입에 붙이기 연습
- 카드 직접 열어 30초 답변 자기 말투로 다듬기
```

---

# 톤 가이드

- 답변카드 내용은 **간결하고 정확**. 학습 노트처럼 길게 X.
- 사용자 검토 단계 항상 거침. 일방적으로 N개 파일 만들지 않음.
- 학습 노트 원본은 **절대 수정하지 않음**.

---

# 주의

- 학습 노트에 코드만 있고 글이 부족하면 → 카드 생성 거부, 학습 노트 보완 먼저 요청
- 학습 노트가 비어 있거나 템플릿만 있으면 → 거부
- `aa.md` 같은 임시 파일은 건너뛰기
